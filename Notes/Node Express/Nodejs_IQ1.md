# 1. What is Node.js?

## Definition

Node.js is an **open-source, cross-platform JavaScript runtime environment** that allows developers to execute JavaScript code outside the browser. It is built on Google's **V8 JavaScript Engine**, which is also used by the Chrome browser.

Before Node.js was introduced, JavaScript could only run inside web browsers. Node.js changed this by enabling JavaScript to run on servers, making it possible to build complete web applications using a single programming language on both the frontend and backend.

Node.js is widely used for:

* Web servers
* REST APIs
* Real-time applications
* Chat applications
* Streaming services
* Microservices
* Serverless functions
* Command-line tools

---

## Official Definition

Node.js is:

> "A JavaScript runtime built on Chrome's V8 JavaScript engine."

This means Node.js provides an environment where JavaScript can interact with the operating system, files, databases, networks, and other server-side resources.

---

## Why Node.js Became Popular

Before Node.js:

* Frontend used JavaScript.
* Backend used Java, PHP, Python, Ruby, C#, etc.

Developers had to learn different languages for frontend and backend.

With Node.js:

* Frontend → JavaScript
* Backend → JavaScript

This simplified development and increased productivity.

---

```text
Application Code

Node.js Runtime

V8 JavaScript Engine

Operating System
```

The V8 Engine converts JavaScript into machine code and executes it efficiently.

# 2. Why was Node.js created?

## Introduction

Node.js was created to solve performance and scalability problems that existed in traditional web servers.

Before Node.js, most web applications used a **thread-per-request model**, where each user request created a separate thread.

This approach worked for small applications but became inefficient when handling thousands of concurrent users.

---

## Problems with Traditional Servers

### Example: PHP/Java Server

When 1000 users connect:

```text
Request 1 → Thread 1
Request 2 → Thread 2
Request 3 → Thread 3
...
Request 1000 → Thread 1000
```

Problems:

* High memory consumption
* Context switching overhead
* Slow performance
* Limited scalability

---

## Ryan Dahl's Vision

Node.js was created in 2009 by:

Ryan Dahl

His goal:

> Create a server capable of handling thousands of simultaneous connections efficiently.

---

## Main Objectives

### 1. Handle Massive Concurrency

Traditional servers:

```text
1 Request = 1 Thread
```

Node.js:

```text
Thousands of Requests = Single Thread + Event Loop
```

This drastically reduces resource consumption.

---

### 2. Eliminate Blocking Operations

Traditional:

```javascript
const data = readFile();
console.log(data);
```

Program waits until file reading completes.

Node.js:

```javascript
readFile((data) => {
   console.log(data);
});
```

Program continues executing.

---

### 3. Real-Time Applications

Applications like:

* Chat apps
* Online games
* Notifications
* Live streaming

Need continuous connections.

Node.js was designed for such workloads.

---

### 4. Use JavaScript Everywhere

Before Node.js:

```text
Frontend → JavaScript
Backend → Java/PHP/Python
```

After Node.js:

```text
Frontend → JavaScript
Backend → JavaScript
```

Single language across the entire stack.

---

## Example Scenario

Suppose 10,000 users are chatting.

Traditional Server:

```text
10,000 Threads
Huge Memory Usage
```

Node.js:

```text
Single Event Loop
Asynchronous Processing
Low Memory Usage
```

This is why chat applications often use Node.js.

---

## Interview Answer

**Node.js was created by Ryan Dahl in 2009 to overcome the scalability limitations of traditional web servers. It introduced an event-driven, non-blocking I/O model that can efficiently handle thousands of concurrent connections with minimal resources, making it ideal for real-time and highly scalable applications.**

# 6. Is Node.js Single-Threaded or Multi-Threaded?

This is one of the most frequently asked Node.js interview questions because many developers incorrectly believe Node.js is either completely single-threaded or completely multi-threaded.

The correct answer is:

> **Node.js is single-threaded for executing JavaScript code but internally uses multiple threads through libuv's thread pool and the operating system.**

---

# Understanding the Concept

When we say Node.js is single-threaded, we mean:

```text
One Main Thread
       ↓
Executes JavaScript Code
```

Only one JavaScript instruction executes at a time.

Example:

```javascript
console.log("A");
console.log("B");
console.log("C");
```

Output:

```text
A
B
C
```

Everything executes sequentially on the main thread.

---

# Then Why Is Node.js So Fast?

Because Node.js doesn't perform heavy operations on the main thread.

Instead:

```text
Main Thread
     ↓
Event Loop
     ↓
libuv Thread Pool
     ↓
Operating System
```

Heavy tasks are delegated elsewhere.

Examples:

* File Reading
* Database Queries
* Network Requests
* DNS Lookup
* Compression
* Encryption

---

# Example

```javascript
const fs = require("fs");

console.log("Start");

fs.readFile("file.txt", () => {
  console.log("File Read");
});

console.log("End");
```

Output:

```text
Start
End
File Read
```

Why?

Because:

```text
readFile()
     ↓
Thread Pool
     ↓
Continue Execution
```

Node.js does not wait.

---

# Node.js Thread Pool

Node.js uses **libuv Thread Pool**.

Default size:

```text
4 Threads
```

Can be increased:

```bash
UV_THREADPOOL_SIZE=8
```

---

# Architecture

```text
                Main Thread
                     ↓
                Event Loop
                     ↓
        -----------------------
        |         |          |
    Thread1   Thread2   Thread3
        |         |          |
      I/O       DNS       Files
```

---

# Single-Threaded Benefits

### 1. Lower Memory Usage

Traditional Servers:

```text
1000 Users
1000 Threads
```

Node.js:

```text
1000 Users
1 Main Thread
```

Consumes significantly less memory.

---

### 2. No Thread Synchronization

In multi-threaded systems:

```java
Thread A
Thread B
Shared Resource
```

Need locks and synchronization.

Node.js avoids most of these issues.

---

### 3. Better Scalability

Can handle thousands of connections simultaneously.

---

# CPU-Intensive Tasks

Single-threaded execution can become a limitation.

Example:

```javascript
while(true){
   // Heavy calculation
}
```

The event loop becomes blocked.

All requests stop.

---

# Solution

Node.js provides:

### Worker Threads

```javascript
const { Worker } = require("worker_threads");
```

Used for:

* Image Processing
* Machine Learning
* Video Encoding
* Large Calculations

---

# Interview Answer

**Node.js is primarily single-threaded because JavaScript code runs on a single main thread managed by the Event Loop. However, Node.js is not purely single-threaded. Internally it uses libuv's thread pool and operating system threads to perform asynchronous operations such as file handling, networking, encryption, and DNS lookups. This hybrid architecture allows Node.js to achieve high scalability and performance.**

```text

     / \
   10  20
```

V8 converts code into a structured tree.

V8 first converts AST into bytecode.

```text

      ↓

      ↓
AST
      ↓
Bytecode
```

# Step 5: Execution

Ignition executes bytecode.

```text

      ↓
Execution
```

---

# Step 6: TurboFan Optimization

Frequently used code is optimized.

```text

    ↓
TurboFan
    ↓
Optimized Machine Code
```

This process is called:

```text

```

(JIT = Just-In-Time Compilation)

```text
      ↓

      ↓

      ↓

      ↓

      ↓

      ↓
```

```text

     ↓
TurboFan Optimization
```

Execution becomes much faster.

# 11. What is Asynchronous Programming in Node.js?

## Definition

Asynchronous Programming is a programming technique where a task starts executing, but the program does **not wait** for that task to finish. Instead, Node.js continues executing other code and processes the result later when the operation completes.

This is one of the core reasons why Node.js is highly scalable and can handle thousands of concurrent requests efficiently.

---

# Why Asynchronous Programming is Needed

Many operations take time:

* Reading files
* Writing files
* Database queries
* API calls
* Network communication
* Sending emails

If Node.js waited for every operation to finish, the application would become slow and unresponsive.

Instead, Node.js uses asynchronous programming.

---

# Real-Life Example

Imagine ordering food in a restaurant.

### Synchronous Approach

```text
Order Food
      ↓
Wait 20 Minutes
      ↓
Receive Food
      ↓
Continue Life
```

Everything stops while waiting.

---

### Asynchronous Approach

```text
Order Food
      ↓
Receive Token
      ↓
Continue Talking
      ↓
Food Arrives Later
```

You continue working while waiting.

Node.js behaves exactly like this.

---

# Example of Asynchronous Programming

```javascript
const fs = require("fs");

console.log("Start");

fs.readFile("data.txt", "utf8", (err, data) => {
    console.log(data);
});

console.log("End");
```

Output:

```text
Start
End
(File Content)
```

---

# Execution Flow

```text

   ↓

   ↓
File Operation Sent to OS
   ↓
Continue Execution
   ↓
End
   ↓
File Completed
   ↓
Callback Executes
```

---

# Asynchronous Programming Using Callbacks

```javascript
function fetchData(callback) {

});

console.log("Program Running");
```

Output:

```text
Program Running
Data Received
```


# Asynchronous Programming Using Async/Await

Modern approach:

```javascript

        setTimeout(() => {
            resolve("Rishi");
        }, 2000);

}

```

Output:

```text

```

### 3. Better Resource Utilization

---

### 4. Improved User Experience

Applications remain responsive.

### 5. Faster APIs

Multiple operations can run concurrently.

```text
    ↓
```

```text

```

Modern solution:

```javascript

Async/Await
```

All JavaScript runs on one thread.

Long-running tasks block other requests.

# 4. Complex Error Handling

Example:

```javascript

   // Sync
```

does not always work for asynchronous callbacks.

# 7. Memory Consumption




```text
Major = 3
Minor = 4
Patch = 7
```

```text
   ↓

```

* Fix API bug
* Fix security issue

No breaking changes.

New features added.

```text
1.2.0
   ↓
1.3.0
```

Examples:

* New endpoint
* New functionality
* Additional options

Backward compatible.

---

# 3. MAJOR Version

Breaking changes.

```text
1.0.0
   ↓
2.0.0
```

Examples:

* Removed features
* Changed API behavior
* Incompatible updates

Requires code modifications.

# 🔄 36. What is the Event Loop?

## Definition

The **Event Loop** is one of the most important concepts in Node.js. It is the mechanism that allows Node.js to perform **non-blocking asynchronous operations** despite JavaScript being **single-threaded**.

In simple words:

> The Event Loop continuously checks whether there are tasks waiting to be executed and moves them to the Call Stack when it becomes empty.

Without the Event Loop, Node.js would process one task at a time and would get stuck whenever a slow operation (database query, API request, file reading, etc.) occurred.

---

## Why Do We Need the Event Loop?

JavaScript executes code on a **single thread**.

That means:

```javascript
console.log("Task 1");
console.log("Task 2");
console.log("Task 3");
```

Output:

```javascript
Task 1
Task 2
Task 3
```

Everything runs sequentially.

Now imagine:

```javascript
const data = fs.readFileSync("largeFile.txt");

console.log(data);
console.log("Finished");
```

Node.js would wait until the file is completely read.

This blocks the entire application.

To solve this problem, Node.js uses:

* Event Loop
* Callback Queue
* Thread Pool (libuv)

This allows expensive tasks to run in the background while JavaScript continues executing other code.

---

## Event Loop Architecture

```text
                ┌───────────────┐
                │   Call Stack  │
                └───────┬───────┘
                        │
                        ▼
                ┌───────────────┐
                │  Event Loop   │
                └───────┬───────┘
                        │
        ┌───────────────┼───────────────┐
        ▼                               ▼
┌─────────────┐                ┌─────────────┐
│ Microtasks  │                │ Macrotasks  │
│ Queue       │                │ Queue       │
└─────────────┘                └─────────────┘
        ▲                               ▲
        │                               │
        └──── Async Operations ─────────┘
```

---

## How It Works

### Step 1: Execute Synchronous Code

```javascript
console.log("Start");

console.log("Middle");

console.log("End");
```

Output:

```javascript
Start
Middle
End
```

All code executes immediately inside the Call Stack.

---

### Step 2: Encounter Async Task

```javascript
console.log("Start");

setTimeout(() => {
    console.log("Timer Done");
}, 2000);

console.log("End");
```

Output:

```javascript
Start
End
Timer Done
```

Explanation:

1. "Start" enters Call Stack.
2. Printed.
3. `setTimeout()` is registered.
4. Timer moves to Web APIs/libuv.
5. "End" prints.
6. After 2 seconds callback enters Queue.
7. Event Loop moves callback to Call Stack.
8. "Timer Done" prints.

---

## Components of Event Loop

### 1. Call Stack

Stores function execution frames.

Example:

```javascript
function first() {
    second();
}

function second() {
    third();
}

function third() {
    console.log("Hello");
}

first();
```

Stack Flow:

```text
first()
 second()
  third()
```

Then unwinds:

```text
third() removed
second() removed
first() removed
```

---

### 2. Callback Queue

Stores completed asynchronous callbacks.

Example:

```javascript
setTimeout(() => {
    console.log("Done");
}, 1000);
```

After 1 second:

```text
Callback Queue
--------------
console.log("Done")
```

Event Loop moves it to the Call Stack when stack becomes empty.

---

### 3. Microtask Queue

# 🔄 39. What is the Callback Queue?

## Definition

The **Callback Queue** (also called the **Task Queue** or **Macrotask Queue**) is a queue where Node.js stores completed asynchronous callback functions before they are executed.

When an asynchronous operation such as:

* `setTimeout()`
* `setInterval()`
* I/O operations
* Network requests
* File system operations

finishes, its callback is placed into the Callback Queue.

The **Event Loop** continuously checks whether the **Call Stack** is empty. If it is empty, the Event Loop moves callbacks from the Callback Queue to the Call Stack for execution.

---

## Why Do We Need the Callback Queue?

JavaScript is single-threaded.

Consider:

```javascript
console.log("Start");

setTimeout(() => {
    console.log("Timer Finished");
}, 2000);

console.log("End");
```

Output:

```javascript
Start
End
Timer Finished
```

If there were no Callback Queue:

* JavaScript would wait 2 seconds.
* Entire application would freeze.

Instead:

1. Timer is registered.
2. JavaScript continues execution.
3. After 2 seconds callback enters Callback Queue.
4. Event Loop moves callback to Call Stack.
5. Callback executes.

```text
      │
      ▼
  Web API/libuv
      │
      ▼

      │
      ▼

      │
      ▼

```

# 🔄 41. What is Callback Hell?

## Definition

**Callback Hell** is a situation where multiple asynchronous operations are nested inside one another, creating deeply indented and difficult-to-maintain code.

It is also known as:

* Pyramid of Doom
* Nested Callbacks
* Callback Nesting

---

## Why Does Callback Hell Happen?

Suppose we need to:

1. Read a file.
2. Process file data.
3. Save processed data.
4. Send email notification.

Using callbacks:

```javascript
readFile("data.txt", function(data) {

    processData(data, function(result) {

        saveData(result, function(savedData) {

            sendEmail(savedData, function(response) {

                console.log(response);

            });

        });

    });

});
```

Notice the pyramid shape:

```text
readFile()
 └── processData()
      └── saveData()
           └── sendEmail()
```

As operations increase, code becomes harder to read.

---

## Real Callback Hell Example

```javascript
setTimeout(() => {

    console.log("Step 1");

    setTimeout(() => {

        console.log("Step 2");

        setTimeout(() => {

            console.log("Step 3");

            setTimeout(() => {

                console.log("Step 4");

            }, 1000);

        }, 1000);

    }, 1000);

}, 1000);
```

Output:

```javascript
Step 1
Step 2
Step 3
Step 4
```

But the code is difficult to manage.

---

## Problems with Callback Hell

### 1. Poor Readability

Code becomes deeply nested.

```javascript
a(function() {
  b(function() {
    c(function() {
      d(function() {
      });
    });
  });
});
```

Hard to understand.

---

### 2. Difficult Debugging

Finding errors becomes complicated.

Example:

```javascript
readFile(file, function(err, data) {

    if(err) {
        // Which callback failed?
    }

});
```

Multiple nested callbacks create confusion.

---

### 3. Difficult Maintenance

Adding new functionality requires editing many nested blocks.

---

### 4. Error Handling Issues

Every callback needs separate error handling.

```javascript
readFile(file, function(err, data) {

    if(err) return console.log(err);

    processData(data, function(err, result) {

        if(err) return console.log(err);

        saveData(result, function(err) {

            if(err) return console.log(err);

        });

    });

});
```

Repeated error checks.

---

## How to Avoid Callback Hell

### Solution 1: Named Functions

Instead of nesting:

```javascript
readFile(file, function(data) {
    processData(data);
});
```

Use:

```javascript
readFile(file, handleFile);

function handleFile(data) {
    processData(data, handleProcess);
}

function handleProcess(result) {
    saveData(result, handleSave);
}

function handleSave(savedData) {
    console.log(savedData);
}
```

Cleaner and more readable.

---

### Solution 2: Promises

```javascript
readFile(file)
    .then(processData)
    .then(saveData)
    .then(sendEmail)
    .catch(console.error);
```

Output flow:

```text
readFile
   ↓
processData
   ↓
saveData
   ↓
sendEmail
```

No pyramid structure.

---

### Solution 3: Async/Await (Recommended)

Modern Node.js applications use `async/await`.

```javascript
async function processFile() {

    try {

        const data = await readFile(file);

        const result = await processData(data);

        const saved = await saveData(result);

        const email = await sendEmail(saved);

        console.log(email);

    } catch(error) {

        console.error(error);

    }
}
```

Benefits:

* Looks like synchronous code
* Easier debugging
* Better readability
* Centralized error handling

---

## Callback Hell vs Promises vs Async/Await

| Feature        | Callback Hell | Promise | Async/Await |
| -------------- | ------------- | ------- | ----------- |
| Readability    | Poor          | Good    | Excellent   |
| Error Handling | Difficult     | Easier  | Best        |
| Nesting        | High          | Low     | None        |
| Maintenance    | Hard          | Easier  | Easy        |
| Modern Usage   | Rare          | Common  | Most Common |

---

## Real Interview Answer

> Callback Hell is a situation where multiple asynchronous callbacks are nested inside one another, creating deeply indented and hard-to-maintain code. It reduces readability, complicates debugging, and makes error handling difficult. Modern JavaScript solves Callback Hell using Promises and async/await, which provide cleaner and more maintainable asynchronous code.

---

## Quick Revision Table

| Concept                        | Description                                   |
| ------------------------------ | --------------------------------------------- |
| Callback Queue                 | FIFO queue storing completed async callbacks  |
| Microtask Queue                | High-priority queue storing Promise callbacks |
| Callback Hell                  | Deeply nested asynchronous callbacks          |
| Queue Priority                 | nextTick → Microtask Queue → Callback Queue   |
| Common Callback Queue Tasks    | setTimeout, setInterval, I/O                  |
| Common Microtasks              | Promise.then, catch, finally                  |
| Best Solution to Callback Hell | Async/Await                                   |
| FIFO                           | First In First Out                            |

# 🔄 42. What are Promises?

## Definition

A **Promise** in JavaScript is an object that represents the **future result of an asynchronous operation**.

In simple words:

> A Promise is a placeholder for a value that will be available later (success or failure).

It is mainly used to handle asynchronous operations like:

* API calls
* File reading
* Database queries
* Timers
* Network requests

---

## Why Promises Were Introduced?

Before Promises, we used callbacks:

```javascript
readFile("file.txt", function(err, data) {
    if(err) {
        console.log(err);
    } else {
        console.log(data);
    }
});
```

Problems:

* Callback hell
* Poor readability
* Difficult error handling

Promises solve this by making async code structured and chainable.

---

## Creating a Promise

```javascript
const promise = new Promise((resolve, reject) => {

    let success = true;

    if (success) {
        resolve("Data fetched successfully");
    } else {
        reject("Error occurred");
    }
});
```

---

## Consuming a Promise

```javascript
promise
    .then((result) => {
        console.log(result);
    })
    .catch((error) => {
        console.log(error);
    });
```

Output:

```text
Data fetched successfully
```

---

## Real Example (API Simulation)

```javascript
function fetchData() {
    return new Promise((resolve, reject) => {

```

# 🔄 43. What are Promise States?

A Promise can be in **one of three states**:

---

## 1. Pending State

> Initial state when the Promise is still running.

```javascript
const p = new Promise(() => {});
```

State:

```text
pending
```

Meaning:

* Operation not completed yet
* Neither success nor failure

---

## 2. Fulfilled State (Resolved)

> Operation completed successfully.

```javascript
const p = new Promise((resolve) => {
    resolve("Success");
});
```

State:

```text
fulfilled
```

Output:

```text
Success
```

---

## 3. Rejected State

> Operation failed.

```javascript
const p = new Promise((resolve, reject) => {
    reject("Error occurred");
});
```

State:

```text
rejected
```

Output:

```text
Error occurred
```

---

## Promise State Diagram

```text
         Pending
            │
   ┌────────┴────────┐
   ▼                 ▼
Fulfilled         Rejected
(resolve)         (reject)
```

> A Promise can change state only once.

```javascript
const p = new Promise((resolve, reject) => {
```

Only first call is considered.

# 🔄 44. What is Promise Chaining?

## Definition

**Promise chaining** is a technique where multiple `.then()` methods are connected to perform a sequence of asynchronous operations.

Each `.then()` receives the result of the previous one.

---

## Why Use Promise Chaining?

Instead of nested callbacks:

```javascript
step1(() => {
    step2(() => {
        step3(() => {});
    });
});
```

We use:

```javascript
step1()
    .then(step2)
    .then(step3);
```

---

## Basic Example

```javascript
const promise = new Promise((resolve) => {
    resolve(10);
});

promise
    .then((value) => {
        console.log(value); // 10
        return value + 5;
    })
    .then((value) => {
        console.log(value); // 15
        return value * 2;
    })
    .then((value) => {
        console.log(value); // 30
    });
```

---

## Execution Flow

```text
10 → 15 → 30
```

Each `.then()` returns a new Promise.

---

## API Example (Real Use Case)

```javascript
fetch("https://api.example.com/user")
    .then(response => response.json())
    .then(user => {
        console.log(user);
        return user.id;
    })
    .then(id => {
        console.log("User ID:", id);
    })
    .catch(error => {
        console.log("Error:", error);
    });
```

---

## Error Handling in Chaining

```javascript
Promise.resolve("Start")
    .then((res) => {
        throw new Error("Something failed");
    })
    .then(() => {
        console.log("This will not run");
    })
    .catch((err) => {
        console.log(err.message);
    });
```

Output:

```text
Something failed
```

---

## Key Feature

> If an error occurs anywhere in the chain, it jumps directly to `.catch()`.

## Promise Chaining Flow

```text
Promise 1 → then → Promise 2 → then → Promise 3 → catch (if error)
```

# 🔄 45. What is async/await?

## Definition

**async/await** is a modern way to handle Promises in JavaScript that makes asynchronous code look and behave like synchronous code.

> `async/await` is built on top of Promises.

---

## Why async/await?

Before:

```javascript
fetchData()
    .then(data => process(data))
    .then(result => save(result));
```

With async/await:

```javascript
const data = await fetchData();
const result = await process(data);
await save(result);
```

Much cleaner and easier to read.

---

## async Function

A function marked with `async` always returns a Promise.

```javascript
async function example() {
    return "Hello";
}

example().then(console.log);
```

Output:

```text
Hello
```

Even though we return a string, it becomes a Promise.

---

## await Keyword

`await` pauses execution until the Promise resolves.

---

## Example

```javascript
function fetchData() {
    return new Promise((resolve) => {
        setTimeout(() => {
            resolve("Data Loaded");
        }, 2000);
    });
}

async function getData() {
    console.log("Start");

    const result = await fetchData();

    console.log(result);

    console.log("End");
}

getData();
```

Output:

```text
Start
Data Loaded
End
```

---

## How It Works Internally

```text
1. async function starts
2. reaches await → pauses execution
3. Promise executes in background
4. Event Loop resumes function
5. continues execution
```

---

## Error Handling in async/await

Use try/catch:

```javascript
async function getData() {
    try {
        const data = await fetchData();
        console.log(data);
    } catch (error) {
        console.log("Error:", error);
    }
}
```

---

## Example with API Call

```javascript
async function getUser() {
    try {
        const response = await fetch("https://api.example.com/user");
        const data = await response.json();

        console.log(data);
    } catch (error) {
        console.log("Failed:", error);
    }
}
```

---

## Sequential vs Parallel Execution

### Sequential (slow)

```javascript
await task1();
await task2();
await task3();
```

---

### Parallel (fast)

```javascript
const p1 = task1();
const p2 = task2();
const p3 = task3();

await Promise.all([p1, p2, p3]);
```

---

## async/await vs Promises

| Feature        | Promises    | async/await       |
| -------------- | ----------- | ----------------- |
| Readability    | Medium      | Excellent         |
| Syntax         | .then chain | Synchronous style |
| Error Handling | .catch      | try/catch         |
| Debugging      | Hard        | Easy              |
| Usage          | Older style | Modern standard   |

Ordering food and tracking status manually:

* Order placed → then → cooking → then → delivered

### async/await

Waiting at restaurant:

* Wait until food is ready
* Then eat step by step

Promise chaining allows sequential execution of async tasks using `.then()`. async/await is a modern syntax built on Promises that allows asynchronous code to be written in a synchronous style using `await`, making it more readable and easier to maintain.

# 🧵 51. What are Worker Threads?

## Definition

**Worker Threads** in Node.js are used to run JavaScript code in **parallel threads**, separate from the main event loop thread.

They were introduced to handle **CPU-intensive tasks** that would otherwise block the Event Loop.

> Node.js is single-threaded for JavaScript execution, but Worker Threads allow true parallel execution of JS.

---

## Why Worker Threads are needed?

Node.js is great for I/O tasks but struggles with CPU-heavy work:

```javascript
// CPU intensive task (bad for main thread)
function heavyTask() {
    let sum = 0;
    for (let i = 0; i < 1e9; i++) {
        sum += i;
    }
    return sum;
}
```

This blocks the event loop.

---

## Worker Threads Example

```javascript
const { Worker, isMainThread, parentPort } = require("worker_threads");

if (isMainThread) {

    const worker = new Worker(__filename);

    worker.on("message", (msg) => {
        console.log("Result:", msg);
    });

} else {

    let sum = 0;

    for (let i = 0; i < 1e9; i++) {
        sum += i;
    }

    parentPort.postMessage(sum);
}
```

---

## Key Features

* Runs JavaScript in parallel threads
* Shares memory using `SharedArrayBuffer`
* Best for CPU-heavy tasks
* Does NOT block Event Loop

---

## Real-Life Analogy

Main thread = cashier
Worker thread = helper doing calculations in background

---

# 🔄 52. What are Child Processes?

## Definition

A **Child Process** allows Node.js to spawn and run **separate system processes** outside the main Node.js process.

It uses OS-level processes (not threads).

## When to use?

* Running scripts
* Heavy isolation tasks
* Running Python/Java programs



## Child Process vs Worker Threads

# 🧩 54. What is Clustering?

## Definition

**Clustering** is a technique in Node.js that allows you to create multiple processes (workers) that run on multiple CPU cores.

> It improves performance by distributing load across CPU cores.

---

## Why clustering is needed?

Node.js uses only **one CPU core by default**.

If system has 8 cores → 7 cores remain unused.

---

## Cluster Example

```javascript
const cluster = require("cluster");
const http = require("http");
const os = require("os");

const numCPUs = os.cpus().length;

if (cluster.isMaster) {

    for (let i = 0; i < numCPUs; i++) {
        cluster.fork();
    }

} else {

    http.createServer((req, res) => {
        res.end("Handled by worker " + process.pid);
    }).listen(3000);
}
```

---

## How Clustering Works

```text
Master Process
   ├── Worker 1 (CPU 1)
   ├── Worker 2 (CPU 2)
   ├── Worker 3 (CPU 3)
   └── Worker N (CPU N)
```

---

## Features

* Uses multiple CPU cores
* Each worker has its own Event Loop
* Improves throughput
* Used in production servers

---

## Real-Life Analogy

One restaurant manager assigns customers to multiple waiters.

---

## Cluster vs Worker Threads

| --------- | --------------- | --------------- |
| Unit      | Process         | Thread          |
| Memory    | Separate        | Shared possible |

# 📦 56. What are modules in Node.js?

In Node.js, **modules are reusable blocks of code** that are isolated from other parts of the application. Each module can contain variables, functions, classes, or objects, and can export them so that other files can use them.

Node.js follows a **modular architecture**, which means every file in Node.js is treated as a separate module by default.

### 🔹 Why modules are used?

* To organize code into smaller, reusable parts
* To avoid code duplication
* To maintain separation of concerns
* To improve maintainability and scalability

### 🔹 Types of modules in Node.js:

1. **Core Modules** (built-in) → e.g. `fs`, `http`, `path`
2. **Local Modules** (user-created)
3. **Third-party Modules** (installed via npm, e.g. `express`)

### 🔹 Example:

```js
// math.js (module)
function add(a, b) {
  return a + b;
}

module.exports = add;
```

```js
// app.js
const add = require('./math');

console.log(add(5, 3)); // 8
```

👉 Here, `math.js` is a module exporting logic, and `app.js` is importing and using it.

# 📦 57. What is module.exports?

`module.exports` is a special object in Node.js used to **export functions, objects, or values from a module** so they can be used in other files.

By default, every file in Node.js has its own `module` object, and `module.exports` is what gets exposed outside the file.

### 🔹 Key idea:

Whatever you assign to `module.exports` becomes the **public API of that module**.

---

### 🔹 Example (export function):

```js
// calculator.js
function multiply(a, b) {
  return a * b;
}

module.exports = multiply;
```

```js
// app.js
const multiply = require('./calculator');

console.log(multiply(4, 5)); // 20
```

---

### 🔹 Exporting multiple values:

```js
// utils.js
function add(a, b) {
  return a + b;
}

function sub(a, b) {
  return a - b;
}

module.exports = {
  add,
  sub
};
```

```js
// app.js
const utils = require('./utils');

console.log(utils.add(10, 5)); // 15
console.log(utils.sub(10, 5)); // 5
```

---

👉 Interview point:

* `module.exports` defines what a module exposes
* It can export a single function or multiple members using an object

# 📦 58. What is require()?

`require()` is a **built-in function in Node.js used to import modules**.

It loads:

* Core modules (like `fs`)
* Local modules (your files)
* External npm packages

---

### 🔹 Syntax:

```js
const moduleName = require('module-name');
```

---

### 🔹 Example (core module):

```js

```

---

### 🔹 Example (local module):

```js

```

### 🔹 How require works internally:

When you call `require()`:

1. Node checks if module is cached
2. If not, it loads the file
3. Wraps it in a function (Module Wrapper)
4. Executes it
5. Returns `module.exports`

### 🔹 Module Wrapper Function:

Every module in Node.js is wrapped like this:

```js
(function (exports, require, module, __filename, __dirname) {
  // module code here
});
```

👉 This is why variables in one file are NOT global.

# 📦 61. What are ES Modules?

**ES Modules (ECMAScript Modules)** are the official standardized module system in JavaScript introduced in ES6. They provide a modern way to organize and reuse code using `import` and `export`.

In Node.js, ES Modules are now fully supported and are considered the **future standard module system**.

---

## 🔹 Key Features of ES Modules:

* Uses `import` and `export` syntax
* Static structure (imports are known at compile time)
* Supports tree-shaking (removing unused code)
* Works in both browser and modern Node.js
* Asynchronous module loading in some environments

---

## 🔹 Example: Named Export

```js id="esmod1"
// math.js
export function add(a, b) {
  return a + b;
}

export function multiply(a, b) {
  return a * b;
}
```

```js id="esmod2"
// app.js
import { add, multiply } from './math.js';

console.log(add(2, 3));       // 5
console.log(multiply(2, 3));  // 6
```

---

## 🔹 Example: Default Export

```js id="esmod3"
// logger.js
export default function log(message) {
  console.log(message);
}
```

```js id="esmod4"
import log from './logger.js';

log("Hello ES Modules");
```

---

## 🔹 How to enable ES Modules in Node.js:

Add this in `package.json`:

```json
{
  "type": "module"
}
```

---

## 🔹 Advantages:

✔ Cleaner syntax
✔ Better optimization (tree-shaking)
✔ Standard across frontend & backend
✔ Future-proof

# 📦 62. What is dependency management?

**Dependency management** refers to the process of managing external libraries (packages) that a Node.js project depends on to function properly.

These dependencies are stored in `package.json` and installed via npm or yarn.

---

## 🔹 Why dependency management is important?

* Ensures project uses correct library versions
* Avoids compatibility issues
* Keeps project reproducible across environments
* Helps manage updates and security patches

---

## 🔹 Types of dependencies:

* dependencies (production)
* devDependencies (development tools)
* peerDependencies (shared dependencies)

---

## 🔹 Example `package.json`:

```json id="dep1"
{
  "name": "my-app",
  "version": "1.0.0",
  "dependencies": {
    "express": "^4.18.0"
  },
  "devDependencies": {
    "nodemon": "^3.0.0"
  }
}
```

---

## 🔹 Example usage:

```bash
npm install express
npm install nodemon --save-dev
```

---

## 🔹 Interview point:

Dependency management ensures:

> “Right package, right version, right environment”

# 📦 63. What are devDependencies?

**devDependencies** are packages that are only required during **development phase**, not in production.

They are installed using:

```bash
npm install <package> --save-dev
```

---

## 🔹 Examples of devDependencies:

* nodemon (auto-restart server)
* eslint (code linting)
* jest (testing framework)
* webpack (bundler)

---

## 🔹 Example:

```bash
npm install nodemon --save-dev
```

```json id="devdep1"
{
  "devDependencies": {
    "nodemon": "^3.0.0"
  }
}
```

---

## 🔹 Why devDependencies are separate?

Because in production:

* We don’t need testing tools
* We don’t need build tools
* We only need runtime dependencies

This reduces:
✔ server size
✔ startup time
✔ unnecessary packages

---

## 🔹 Example scenario:

* Development: use `nodemon` for auto-restart
* Production: use `node server.js`

---

## 🔹 Interview line:

> devDependencies are tools used only for development and are not required in production runtime.

# 📦 64. What are peerDependencies?

**peerDependencies** are dependencies that a package expects the *project using it* to already have installed.

They are commonly used when building **libraries or plugins**.

---

## 🔹 Why peerDependencies exist?

To avoid:

* Multiple versions of same library
* Conflicts in shared dependencies

---

## 🔹 Example scenario:

Imagine you are building a React UI library:

```json id="peer1"
{
  "peerDependencies": {
    "react": "^18.0.0"
  }
}
```

👉 This means:

> “My library works with React, but I expect the user to already have React installed.”

---

## 🔹 Example:

If you install a UI library:

```bash
npm install my-ui-library
```

But React is not installed:
npm will warn you:

> unmet peer dependency: react

---

## 🔹 Key characteristics:

* Not automatically installed (in older npm versions)
* Must be installed by the project using the package
* Ensures compatibility between packages

---

## 🔹 Real-world examples:

* React libraries (MUI, Ant Design)
* Plugin-based systems
* Angular packages

---

## 🔹 Interview line:

> peerDependencies define packages that must be installed in the host project to ensure compatibility.

# 📦 66. What is npm update?

**`npm update`** is a command used to **update installed packages in a Node.js project to their latest compatible versions** based on version ranges defined in `package.json`.

---

## 🔹 How it works:

* Reads `package.json`
* Checks installed packages in `node_modules`
* Updates packages within allowed semver range (`^` or `~`)
* Updates `package-lock.json`

---

## 🔹 Example `package.json`:

```json id="upd1"
{
  "dependencies": {
    "express": "^4.18.0"
  }
}
```

👉 This means:

* npm can update Express to any `4.x.x` version but not `5.x.x`

---

## 🔹 Command:

```bash
npm update
```

---

## 🔹 Update a specific package:

```bash
npm update express
```

---

## 🔹 Important note:

* It does NOT upgrade to breaking versions automatically
* For major upgrades, use:

```bash
npm install express@latest
```

---

## 🔹 Interview line:

> npm update upgrades installed packages within the allowed version range defined in package.json without breaking compatibility.

**`npm audit`** is a security feature in npm used to **analyze project dependencies and detect known vulnerabilities**.

It scans:

* Direct dependencies
* Transitive dependencies (nested packages)

---

## 🔹 Command:

```bash
npm audit
```

## 🔹 What it shows:

* Affected packages
* Fix suggestions

## 🔹 Example output idea:

```
found 3 vulnerabilities (2 moderate, 1 high)
```

---

## 🔹 Auto fix command:

```bash
npm audit fix
```

This automatically:

* Updates vulnerable packages
* Installs safe versions

## 🔹 Force fix (risky):

```bash
npm audit fix --force
```

👉 May introduce breaking changes

---

## 🔹 Why npm audit is important:

* Prevents security risks in production
* Helps maintain secure dependencies
* Used in CI/CD pipelines

---

## 🔹 Interview line:

> npm audit is a built-in security tool that scans project dependencies for vulnerabilities and suggests fixes.

# 📦 68. What is package publishing?

**Package publishing** is the process of **uploading your Node.js package to npm registry** so that others can install and use it.

---

## 🔹 Steps to publish a package:

### 1. Initialize project:

```bash
npm init -y
```

---

### 2. Create module:

```js id="pub1"
// index.js
function greet(name) {
  return `Hello ${name}`;
}

module.exports = greet;
```

---

### 3. Login to npm:

```bash
npm login
```

---

### 4. Publish package:

```bash
npm publish
```

---

## 🔹 Example `package.json`:

```json id="pub2"
{
  "name": "my-greet-package",
  "version": "1.0.0",
  "main": "index.js"
}
```

---

## 🔹 Install published package:

```bash
npm install my-greet-package
```

---

## 🔹 Important rules:

* Package name must be unique
* Version must follow semver (1.0.0, 1.0.1, etc.)
* Cannot publish same version twice

---

## 🔹 Interview line:

> Package publishing is the process of sharing reusable Node.js code by uploading it to the npm registry for public or private use.

# 📦 69. What is a private package?

A **private package** is an npm package that is **not publicly accessible on the npm registry** and can only be used within a specific organization or user account.

---

## 🔹 Why private packages are used:

* Internal company libraries
* Confidential business logic
* Shared enterprise tools
* Secure reusable components

---

## 🔹 How to create private package:

In `package.json`:

```json id="priv1"
{
  "name": "@mycompany/utils",
  "version": "1.0.0",
  "private": true
}
```

OR for npm organization:

```json
{
  "name": "@company/utils",
  "version": "1.0.0",
  "publishConfig": {
    "access": "restricted"
  }
}
```

---

## 🔹 Install private package:

```bash
npm install @company/utils
```

---

## 🔹 Key features:

* Not visible publicly on npm
* Requires authentication
* Used in enterprise environments

---

## 🔹 Difference from public package:

| Feature    | Public Package | Private Package   |
| ---------- | -------------- | ----------------- |
| Access     | Anyone         | Restricted        |
| Visibility | npm registry   | Organization only |
| Security   | Low            | High              |

---

## 🔹 Interview line:

> Private packages are npm packages restricted to specific users or organizations, mainly used for internal development.

# 📦 70. What are npm scripts?

**npm scripts** are predefined commands written inside `package.json` that allow you to automate tasks like running servers, tests, builds, etc.

---

## 🔹 Location:

```json id="script1"
{
  "scripts": {
    "start": "node app.js",
    "dev": "nodemon app.js",
    "test": "jest"
  }
}
```

---

## 🔹 Run scripts:

```bash
npm start
npm run dev
npm test
```

---

## 🔹 Important note:

* `start` and `test` can run without `run`
* Other scripts require `npm run`

---

## 🔹 Example workflow:

```json id="script2"
{
  "scripts": {
    "start": "node server.js",
    "dev": "nodemon server.js",
    "build": "webpack",
    "lint": "eslint ."
  }
}
```

---

## 🔹 Advanced chaining:

```json id="script3"
{
  "scripts": {
    "build": "npm run lint && webpack",
    "prod": "npm run build && node server.js"
  }
}
```

---

## 🔹 Why npm scripts are useful:

* Automate repetitive tasks
* Standardize commands across teams
* Improve productivity
* Integrate tools easily

---

## 🔹 Interview line:

> npm scripts are automation commands defined in package.json that help run, build, test, and manage Node.js applications efficiently.
