# 📘 1. JavaScript Fundamentals (1–20)

# 1. What is JavaScript?

## Interview Definition

**JavaScript (JS)** is a high-level, interpreted, dynamically typed, object-oriented, and event-driven programming language primarily used to create interactive and dynamic web applications.

Initially, JavaScript was developed by **Brendan Eich in 1995** at Netscape and was originally called **Mocha**, then **LiveScript**, and finally **JavaScript**.

Today, JavaScript is used not only in browsers but also on servers (Node.js), mobile apps (React Native), desktop apps (Electron), game development, and even IoT devices.

---

## GFG-Style Explanation

Think of a website as:

* HTML = Structure (Skeleton)
* CSS = Styling (Clothes)
* JavaScript = Behavior (Brain)

Without JavaScript:

* Buttons cannot respond
* Forms cannot validate instantly
* Dynamic content cannot update
* APIs cannot be called

JavaScript makes web pages interactive.

Example:

```html
<button onclick="showMessage()">
  Click Me
</button>

<script>
function showMessage() {
  alert("Hello User!");
}
</script>
```

When the user clicks the button, JavaScript executes the function.

---

# Internal Working of JavaScript

JavaScript runs inside a **JavaScript Engine**.

Examples:

* Chrome → V8 Engine
* Firefox → SpiderMonkey
* Safari → JavaScriptCore

### Execution Process

```text
JavaScript Code
       ↓
Parser
       ↓
Abstract Syntax Tree (AST)
       ↓
Interpreter
       ↓
Bytecode
       ↓
JIT Compiler
       ↓
Machine Code
```

Modern engines use **Just-In-Time (JIT) Compilation** to improve performance.

## Definition

JavaScript is called a **scripting language** because it was originally designed to write scripts that automate tasks inside web browsers.

A script is a set of instructions executed at runtime without needing traditional compilation.

---

## What is a Script?

A script is code that:

* Automates tasks
* Runs inside another program
* Is interpreted during execution

Example:

```js
alert("Welcome!");
```

Browser reads and executes immediately.

---

# Why Was JavaScript Created?

Before JavaScript:

* Websites were static.
* Any validation required server requests.

Example:

```text

```

This was slow.

JavaScript solved this:

```js

  alert("Invalid Email");
```

Validation happens instantly.

# 5. What is the Difference Between JavaScript and ECMAScript?

This is one of the most common JavaScript interview questions, especially in MERN Stack interviews.

---

# Simple Definition

Many beginners think JavaScript and ECMAScript are the same thing.

They are related but not identical.

### ECMAScript

ECMAScript is a **standard (set of rules and specifications)** that defines how a scripting language should work.

### JavaScript

JavaScript is the **actual programming language** that implements the ECMAScript standard.

Think of it like:

```text
ECMAScript = Blueprint / Rules

JavaScript = Actual Building
```

or

```text
ECMAScript = Recipe

JavaScript = Final Dish
```

---

# Why ECMAScript Was Created?

In the 1990s different browsers implemented JavaScript differently.

Problems:

```text
Code works in Browser A
Code fails in Browser B
```

To solve this issue, ECMA International created a standard called:

```text
ECMAScript
```

Now browser vendors follow the same specification.

---

# Real-World Analogy

Suppose the government creates traffic rules.

```text
Traffic Rules = ECMAScript

Cars = JavaScript Engines
```

Every car manufacturer follows those rules.

Similarly:

```text
Chrome V8
Firefox SpiderMonkey
Safari JavaScriptCore
```

all implement ECMAScript standards.

# 23. What is Variable Hoisting?

## Definition

Hoisting is JavaScript’s behavior of moving variable and function declarations to the top of their scope before execution.

This happens during the **memory creation phase** of the execution context.

---

# Example with var

```javascript

var x = 10;
```

Internally behaves like:

```javascript
var x;
console.log(x);
x = 10;
```

Output:

```javascript
undefined
```

---

# Why?

During memory creation:

* `x` is created
* Assigned `undefined`

During execution:

* Value becomes `10`

---

# Hoisting with let

```javascript
console.log(a);
let a = 5;
```

Output:

```javascript
ReferenceError
```

---

# Why?

Because `let` is hoisted but placed inside **Temporal Dead Zone (TDZ)**.

---

# Hoisting with const

```javascript
console.log(pi);
const pi = 3.14;
```

Error.

Same reason:

* Hoisted
* Inside TDZ

---

# Temporal Dead Zone (TDZ)

The time between:

* Variable declaration being hoisted
  AND
* Variable initialization

Example:

```javascript
{
   // TDZ starts
   let x = 10;
   // TDZ ends
}
```

Access before initialization → error.

---

# Internal Working

JavaScript execution phases:

### 1. Memory Creation Phase

* Variables stored in memory
* Functions fully stored

### 2. Execution Phase

* Code runs line by line

---

Example:

```javascript
var x = 5;
function greet() {
   console.log("Hello");
}
```

Memory phase:

* x → undefined
* greet → full function

---

# Real-World Importance

* Reference errors
* Scope confusion

# Advantages

# 29. What is Lexical Scope?

## Definition

Lexical scope means that the accessibility of variables is determined by **where variables and functions are written in code (their physical location)**.

In simple words:
A function can access variables from:

* Its own scope
* Parent scope
* Global scope

This scope relationship is decided during code writing, not during execution.

That’s why it is called **lexical** (based on code structure).

---

# Basic Example

```javascript
let globalVar = "I am global";

function outer() {
    let outerVar = "I am outer";

    function inner() {
        let innerVar = "I am inner";

        console.log(globalVar);
        console.log(outerVar);
        console.log(innerVar);
    }

    inner();
}

outer();
```

Output:

```javascript
I am global
I am outer
I am inner
```

---

# How It Works Internally

JavaScript uses a **scope chain**.

When a variable is accessed:

1. JS checks current scope
2. If not found → checks parent scope
3. Continues until global scope
4. If still not found → ReferenceError

Example:

```javascript
let a = 10;

function parent() {
    let b = 20;

    function child() {
        let c = 30;
        console.log(a, b, c);
    }

    child();
}

parent();
```

Search order for variables inside `child()`:

* Check child scope
* Check parent scope
* Check global scope

---

# Important Rule

Child functions can access parent variables.

But parent functions cannot access child variables.

Example:

```javascript
function outer() {
    function inner() {
        let x = 100;
    }

    console.log(x);
}
```

Output:

```javascript
ReferenceError
```

---

# Lexical Scope + Closures

Lexical scope is the foundation of closures.

Example:

```javascript
function outer() {
    let count = 0;

    return function inner() {
        count++;
        console.log(count);
    };
}

const counter = outer();

counter();
counter();
```

Output:

```javascript
1
2
```

Why?
Because inner function remembers outer scope.

---

# Real-World MERN Example

Middleware example:

```javascript
function authorize(role) {
    return function(req, res, next) {
        if (req.user.role === role) {
            next();
        }
    };
}
```

Inner middleware accesses outer `role`.

* Variable `count` created
* `inner()` created

When `outer()` returns:

* Normally memory destroyed

But:

* Since returned function still needs `count`
* JS keeps lexical environment alive

This creates closure.

# 38. What is Scope Chaining?



---

# How Scope Chain Works

Inside `inner()`:

* Search for `c` → found in inner scope
* Search for `b` → found in outer scope
* Search for `a` → found in global scope

This creates a chain.

---


Because `x` not found in scope chain.

* Current variable references


# 45. What is the Difference Between Arrow Functions and Regular Functions?

This is one of the most commonly asked JavaScript interview questions.

---

# Definition

## Regular Function

Traditional way of defining functions using `function` keyword.

```javascript
function greet() {
    console.log("Hello");
}
```

---

## Arrow Function

Introduced in ES6 with shorter syntax.

```javascript
const greet = () => {
    console.log("Hello");
}
```

---

# Key Differences

| Feature            | Regular Function              | Arrow Function           |
| ------------------ | ----------------------------- | ------------------------ |
| Syntax             | Longer                        | Shorter                  |
| `this` binding     | Own `this`                    | Lexical `this`           |
| `arguments` object | Available                     | Not available            |
| Constructor        | Can be used                   | Cannot be used           |
| Hoisting           | Function declarations hoisted | Not fully hoisted        |
| Best for           | Object methods, constructors  | Callbacks, array methods |

---

# 1. Difference in Syntax

Regular:

```javascript
function add(a, b) {
    return a + b;
}
```

Arrow:

```javascript
const add = (a, b) => a + b;
```

Arrow syntax is shorter.

---

# 2. Difference in `this`

This is the biggest difference.

---

## Regular Function → Own `this`

```javascript
const user = {
    name: "Rishi",
    show: function() {
        console.log(this.name);
    }
};

user.show();
```

Output:

```javascript
Rishi
```

Here `this` refers to `user`.

---

## Arrow Function → Lexical `this`

```javascript
const user = {
    name: "Rishi",
    show: () => {
        console.log(this.name);
    }
};

user.show();
```

Output:

```javascript
undefined
```

Why?

Arrow functions do not create their own `this`.
They inherit `this` from surrounding scope.

---

# 3. `arguments` Object

Regular function:

```javascript
function test() {
    console.log(arguments);
}
test(1, 2, 3);
```

Works.

---

Arrow function:

```javascript
const test = () => {
    console.log(arguments);
};
```

Error.

Arrow functions don’t have `arguments`.

Use rest operator:

```javascript
const test = (...args) => {
    console.log(args);
}
```

---

# 4. Constructor Behavior

Regular functions can be constructors.

```javascript
function Person(name) {
    this.name = name;
}

const p = new Person("Rishi");
```

Works.

---

Arrow functions cannot.

```javascript
const Person = (name) => {
    this.name = name;
};

new Person("Rishi");
```

Error.

---

# Real-World MERN Example

React callback:

```javascript
const handleClick = () => {
   console.log("Clicked");
}
```

Express route handler:

```javascript
function getUsers(req, res) {
    res.send("Users");
}
```

---

# When to Use Which?

Use Regular Functions:
✔ Object methods
✔ Constructors
✔ Functions needing dynamic `this`

Use Arrow Functions:
✔ Callbacks
✔ Promises
✔ Array methods
✔ React handlers

# Common Interview Questions

* Why arrow functions don’t have `this`?
* Can arrow functions be constructors?
* Why avoid arrow functions in object methods?

---

# Summary

Regular functions have their own `this`, `arguments`, and constructor behavior. Arrow functions are shorter and inherit `this` from outer scope.

# 49. What is Recursion?

## Definition

**Recursion** is a programming technique where a function calls itself until a stopping condition (base condition) is met.

Simple definition:

> A function solving a problem by calling itself on smaller subproblems.

---

# Basic Structure of Recursion

A recursive function has two parts:

### 1. Base Condition

Stops recursion.

### 2. Recursive Call

Function calls itself.

```javascript
function recursiveFunction() {
   if(baseCondition) return;
   recursiveFunction();
}
```

---

# Example 1: Factorial

Mathematical formula:

```text
5! = 5 × 4 × 3 × 2 × 1
```

Recursive definition:

```text
n! = n × (n-1)!
```

Code:

```javascript
function factorial(n) {
    if (n === 1) {
        return 1; // base case
    }
    return n * factorial(n - 1);
}

console.log(factorial(5));
```

Output:

```javascript
120
```

---

# Step-by-Step Execution

```javascript
factorial(5)
→ 5 * factorial(4)
→ 5 * 4 * factorial(3)
→ 5 * 4 * 3 * factorial(2)
→ 5 * 4 * 3 * 2 * factorial(1)
→ 5 * 4 * 3 * 2 * 1
= 120
```

---

# Example 2: Fibonacci

```javascript
function fibonacci(n) {
    if (n <= 1) return n;

    return fibonacci(n - 1) + fibonacci(n - 2);
}

console.log(fibonacci(6));
```

Output:

```javascript
8
```

---

# Internal Working (Call Stack)

Every recursive call gets pushed into the **call stack**.

Example:

```javascript
function countDown(n) {
    if (n === 0) return;
    console.log(n);
    countDown(n - 1);
}

countDown(3);
```

Stack:

```text
countDown(3)
countDown(2)
countDown(1)
countDown(0)
```

After base case, stack starts popping.

---

# Real-World Use Cases

Recursion is used in:

* Tree traversal
* DFS algorithms
* File system navigation
* JSON parsing
* DOM traversal

Example:

```javascript
function printComments(comments) {
    comments.forEach(comment => {
        console.log(comment.text);
        if(comment.replies) {
            printComments(comment.replies);
        }
    });
}
```

---

# Advantages

✔ Elegant solution
✔ Good for tree/graph problems
✔ Cleaner for divide-and-conquer

---

# Limitations

✖ Stack overflow risk
✖ Slower than iteration sometimes
✖ Hard to debug

---

# Common Interview Questions

* Difference between recursion and iteration?
* What is base condition?
* What causes stack overflow?

---

# Summary

Recursion is when a function calls itself until a base condition stops execution.

# 50. What is Function Currying?

## Definition

**Currying** transforms a function with multiple arguments into a sequence of functions where each function takes one argument.

Normal function:

```javascript
f(a, b, c)
```

Curried function:

```javascript
f(a)(b)(c)
```

---

# Example

Normal function:

```javascript
function add(a, b, c) {
    return a + b + c;
}

console.log(add(1, 2, 3));
```

Output:

```javascript
6
```

Curried version:

```javascript
function curryAdd(a) {
    return function(b) {
        return function(c) {
            return a + b + c;
        };
    };
}

console.log(curryAdd(1)(2)(3));
```

Output:

```javascript
6
```

---

# Arrow Function Version

```javascript
const add = a => b => c => a + b + c;

console.log(add(1)(2)(3));
```

---

# Internal Working

```javascript
add(1)
```

Returns:

```javascript
function(b) { ... }
```

Then:

```javascript
add(1)(2)
```

Returns:

```javascript
function(c) { ... }
```

Finally:

```javascript
add(1)(2)(3)
```

Returns final value.

---

# Why Use Currying?

Useful for:

* Reusability
* Functional programming
* Configurable functions

---

# Real-World Example

```javascript
const multiply = a => b => a * b;

const double = multiply(2);
const triple = multiply(3);

console.log(double(5)); // 10
console.log(triple(5)); // 15
```

# React Example

```javascript
    console.log(id);
};
```

# Advantages

✔ Functional programming

# Limitations

✖ Too much nesting

# Summary

Currying converts a function with multiple parameters into nested functions with one parameter each.

# 51. What is Partial Application?

Very commonly confused with currying.

---

# Definition

**Partial application** means fixing a few arguments of a function and returning a new function.

Example:

```javascript

```

After partial application:

```javascript

```

Where `a` is fixed.

---

# Example

Normal function:

```javascript

    return a * b;
}
```

Partial application:

```javascript

    return function(b) {
        return multiply(a, b);
    };
}

const double = partialMultiply(2);

console.log(double(5));
```

Output:

```javascript
10
```

---

# Currying vs Partial Application

This is a favorite interview question.

| ---------------------- | ------------------------- | ------------------- |
| Converts function      | Yes                       | No                  |
| Arguments per function | One                       | Multiple possible   |
| Purpose                | Break function into chain | Fix some arguments  |



# Real-World Example

# Definition

An **async function** is a function declared with `async` keyword that always returns a Promise.

Simple definition:

> Async functions simplify working with Promises using `await`.

---

# Syntax

```javascript id="24azdz"
async function demo() {}
```

---

# Example

```javascript id="xqtxgi"
async function greet() {
    return "Hello";
}
```

Return value becomes Promise.

```javascript id="70v4wp"
greet().then(console.log);
```

Output:

```javascript id="p0uyt5"
Hello
```

Equivalent to:

```javascript id="4a5gh4"
function greet() {
    return Promise.resolve("Hello");
}
```

---

# Using Await

```javascript id="egnn41"
async function fetchData() {
    let result = await Promise.resolve("Data loaded");
    console.log(result);
}

fetchData();
```

Output:

```javascript id="8ezw0e"
Data loaded
```

---

# Internal Working

When JS sees:

```javascript id="m7hm5u"
await promise
```

It:

1. Pauses function execution
2. Waits for promise resolution
3. Resumes execution

Non-blocking behavior maintained.

---

# Without Async/Await

Using promises:

```javascript id="8ho94r"
fetchData()
   .then(data => process(data))
   .catch(err => console.log(err));
```

---

# With Async/Await

```javascript id="6ig7hv"
try {
   const data = await fetchData();
   process(data);
} catch(err) {
   console.log(err);
}
```

Cleaner.

---

# Real-World MERN Example

API request:

```javascript id="kgfls1"
async function getUsers() {
    const users = await User.find();
    return users;
}
```

Express route:

```javascript id="5yd3lc"
app.get("/users", async (req, res) => {
    const users = await User.find();
    res.json(users);
});
```

---

# Advantages

✔ Cleaner async code
✔ Better readability
✔ Easy error handling

---

# Limitations

✖ Can hide promise behavior from beginners
✖ Must handle errors properly

Async/await is syntax built on Promises.

Async functions simplify Promise handling using `async` and `await`.

# 63. What is Function Borrowing?

Important `this` interview topic.

---

# Definition

Function borrowing means borrowing a method from one object and using it with another object.

Usually done using:

* `call()`
* `apply()`
* `bind()`

Simple definition:

> One object uses another object's method.

---

# Example

```javascript id="gtrm7k"
const person1 = {
    name: "Rishi",
    greet: function() {
        console.log("Hello " + this.name);
    }
};

const person2 = {
    name: "Aman"
};
```

Borrow method:

```javascript id="i0h6g4"
person1.greet.call(person2);
```

Output:

```javascript id="61hw3e"
Hello Aman
```

---

# Internal Working

Normally:

```javascript id="kkw2rx"
person1.greet()
```

`this = person1`

Using call:

```javascript id="58ttxp"
person1.greet.call(person2)
```

Now:
`this = person2`

---

# Example with Arguments

```javascript id="tmyui7"
const user = {
    fullName(city) {
        console.log(this.name + " " + city);
    }
};

const obj = { name: "Rishi" };

user.fullName.call(obj, "Indore");
```

Output:

```javascript id="3v1m9w"
Rishi Indore
```

---

# Real-World MERN Example

Shared utility methods between objects.

---

# Advantages

✔ Code reuse
✔ Reduces duplication
✔ Powerful `this` control

---

# Limitations

✖ Confusing for beginners
✖ Requires strong understanding of `this`

---

# Common Interview Questions

* What is function borrowing?
* Difference between call/apply/bind?

---

# Summary

Function borrowing lets one object reuse another object’s function using call/apply/bind.

# 64. What are Factory Functions?

Very important object creation pattern.

---

# Definition

A **factory function** is a function that returns an object.

Simple definition:

> Function used to create and return objects.

---

# Example

```javascript id="g2k1j0"
function createUser(name, age) {
    return {
        name,
        age,
        greet() {
            console.log("Hello " + name);
        }
    };
}
```

Usage:

```javascript id="0bwl2h"
const user1 = createUser("Rishi", 22);
const user2 = createUser("Aman", 24);
```

---

# Internal Working

Call:

```javascript id="rdxv72"
createUser("Rishi", 22)
```

Function creates:

* Object
* Properties
* Methods

Returns object.

---

# Why Use Factory Functions?

Without factory function:

```javascript id="cwnp4s"
const user = {
   name: "Rishi",
   age: 22
};
```

Repeated creation becomes messy.

Factory solves this.

---

# Real-World MERN Example

Creating reusable API response objects.

```javascript id="3sy4jc"
function createResponse(success, data) {
    return {
        success,
        data
    };
}
```

---

# Advantages

✔ Easy object creation
✔ Cleaner code
✔ Flexible

---

# Limitations

✖ Methods recreated for each object
✖ Less memory efficient than prototype-based approach

---

# Common Interview Questions

* Factory vs constructor function?
* Why use factory functions?

---

# Summary

Factory functions create and return objects dynamically.

# 65. What are Constructor Functions?

Very important for JavaScript OOP.

---

# Definition

A **constructor function** is a regular function used with `new` keyword to create objects.

Simple definition:

> Constructor functions act like object blueprints.

---

# Syntax

```javascript id="q0j2yb"
function Person(name, age) {
    this.name = name;
    this.age = age;
}
```

Usage:

```javascript id="0z4s8z"
const p1 = new Person("Rishi", 22);
```

---

# Internal Working of `new`

When:

```javascript id="vl8w98"
new Person("Rishi", 22)
```

JavaScript does:

### Step 1

Creates empty object.

```javascript id="rjyqq4"
{}
```

### Step 2

Sets `this` to new object.

### Step 3

Executes constructor code.

```javascript id="q6ggbd"
this.name = "Rishi"
this.age = 22
```

### Step 4

Returns object automatically.

---

# Example with Methods

```javascript id="i0qhlv"
function Person(name) {
    this.name = name;

    this.greet = function() {
        console.log("Hello " + this.name);
    };
}

const p1 = new Person("Rishi");
p1.greet();
```

---

# Better with Prototype

```javascript id="gnx2ak"
function Person(name) {
    this.name = name;
}

Person.prototype.greet = function() {
    console.log("Hello " + this.name);
};
```

More memory efficient.

---

# Constructor vs Factory Function

| Feature           | Constructor | Factory    |
| ----------------- | ----------- | ---------- |
| Uses new          | Yes         | No         |
| Uses this         | Yes         | Usually No |
| Prototype support | Yes         | Limited    |

---

# Real-World MERN Example

Used less in modern React, but still important for JS fundamentals.

---

# Advantages

✔ Supports prototypes
✔ Memory efficient
✔ OOP friendly

---

# Limitations

✖ Requires `new`
✖ `this` confusion possible
